Data Flow
How information moves through Claude Code — from user input to model response to tool execution and back.

Primary conversation loop

Step 1
User types message
PromptInput captures keystrokes. On submit, message enters AppStateStore.
Step 2
QueryEngine.runQuery()
Wraps message with system prompt (from context.ts), file history, conversation history.
Step 3
query.ts → Anthropic API
Streaming call to the configured model. Tokens stream back in real-time.
Step 4
Tool call intercepted
If model outputs a tool_use block, query.ts pauses the stream and invokes the tool.
Step 5
Permission check
tool.checkPermissions() fires → allow / deny / ask. If ask: dialog shown to user.
Step 6
Tool executes
Bash runs command, FileEditTool patches file, AgentTool launches sub-QueryEngine, etc.
Step 7
Result fed back
tool_result block appended to conversation. Loop returns to Step 3 until no more tool calls.
Step 8
Response displayed
Messages.tsx renders final text. Conversation saved to session transcript.

Why Bun over Node?
Startup time is critical for a CLI tool. Bun's sub-50ms cold start vs Node's 200ms+ is a meaningful UX win. Built-in TypeScript support also removes a build step.
Core Engine
Startup & REPL
How Claude Code initializes, what happens before the first prompt, and the two execution modes.

main.tsx — CLI entrypoint

main.tsx is the first file executed. It:

  1. Parses CLI arguments via Commander.js
  2. Runs startup tasks in parallel: MDM policy read, keychain API key prefetch, GrowthBook feature flag init
  3. Determines execution mode: REPL (interactive) or headless (pipe/script)
  4. Mounts the React/Ink root or runs the headless loop

Execution modes

REPL Mode
Interactive terminal session. React/Ink renders the UI. User types, model responds. Default when run without piped input.
Interactive
Headless Mode
Pipe input, scripted automation, SDK usage. No Ink UI. Outputs to stdout/stderr. Used by CI, scripts, the SDK.
Automation

QueryEngine.ts vs query.ts

FileRoleUsed by
query.tsRaw API loop — sends messages, handles streaming, tool call interception, auto-compactQueryEngine
QueryEngine.tsHigh-level wrapper — manages conversation state, file history snapshots, session storage, plugin loadingREPL, SDK, AgentTool

context.ts — System prompt

Before every turn, context.ts builds the system prompt by reading:

  • Git status of current repo
  • CLAUDE.md files (project root + parent dirs + .claude/ subdir)
  • Memory files from ~/.claude/projects/…/memory/
  • Active tool list, permission mode, current directory
Product Impact
CLAUDE.md files are how teams customize Claude Code per-project — shared instructions, coding standards, tool restrictions. Key enterprise adoption surface.
Core Engine
Query Loop
The heart of Claude Code — the message/response/tool cycle that drives every interaction.

What query.ts does

  • Streaming — tokens from the Anthropic API stream in real-time, displayed as they arrive
  • Tool call interception — when the model emits tool_use, stream pauses, tool executes, result fed back
  • Auto-compact — monitors token count; when context gets too long, old messages are summarized
  • Retry logic — handles API errors, rate limits, network issues

The agentic loop

while (true) {
  const response = await anthropicAPI.stream(messages, tools)

  if (response.hasToolUse) {
    for (const toolCall of response.toolCalls) {
      const permission = await tool.checkPermissions(toolCall)
      if (permission === 'deny') continue
      if (permission === 'ask') await promptUser()

      const result = await tool.execute(toolCall)
      messages.push({ role: 'tool', content: result })
    }
    continue  // ← loop back, feed results to model
  }

  break  // ← no more tool calls, done
}

Auto-compact

When token count approaches limits, services/compact/ summarizes old messages:

  • Triggered automatically in query.ts
  • Fires PreCompact and PostCompact lifecycle hooks
  • Summary injected as a system message at the start of truncated history
  • User can also trigger manually with /compact
Token cost note for PMs
Each tool result is a full API round-trip. A task with 10 tool calls = 10 API calls (but streamed in one session). Large file reads dramatically increase token usage.

Rule syntax

{
  "permissions": {
    "allow": [
      "Bash(git:*)",          // all git subcommands
      "Bash(npm run test)",   // exact command
      "Read",                 // all Read operations
      "Edit(.claude)"         // files in .claude/
    ],
    "deny": ["Bash(rm -rf:*)"],
    "defaultMode": "acceptEdits"
  }
}
Enterprise surface
MDM policy settings take highest precedence and cannot be overridden by users or project config. This is how enterprise IT enforces Claude Code restrictions fleet-wide.

State & React Hooks
How runtime state is managed — AppStateStore, React context, and key hooks.

AppStateStore.ts

Custom Zustand-like store at state/AppStateStore.ts. Holds all runtime state:

  • messages — current conversation messages
  • permission context — current mode, session decisions
  • tool state — in-progress tool calls, results
  • tasks — active async task list
  • session info — session ID, model, config

Key React hooks

HookPurpose
useCanUseToolPermission check — returns whether a tool can be used right now
useSettings / useSettingsChangeRead settings + react to settings file changes
useTasksV2Task management — create, update, poll task state
useMainLoopModelGet/set the active model (sonnet/opus/haiku)
useMergedToolsCombines built-in + MCP + plugin tools into one list
useMergedCommandsCombines built-in slash commands + plugin commands
useVoiceVoice STT input integration
toolPermission/Permission dialog hooks — show/hide prompts, record decisions
Architecture note
The split between state/ (plain store) and hooks/ (React bindings) is deliberate — the store can be used headlessly (SDK, tests) without React.

Settings System
The 4-layer cascading settings system — how configuration is loaded, merged, and applied.

4 layers, in override order

Layer 1 (lowest)
User settings
~/.claude/settings.json — global preferences, applies to all projects
Layer 2
Project settings
.claude/settings.json — committed to repo, team-shared. Overrides user settings.
Layer 3
Local settings
.claude/settings.local.json — gitignored personal overrides for this project
Layer 4 (highest)
Policy/managed settings
MDM enterprise policy — cannot be overridden by any user or project config

CLAUDE.md = natural language instructions in the system prompt (how to write code, project conventions).

Use cases for hooks
Auto-formatting, logging bash commands, blocking dangerous patterns, running tests after code changes, notifying Slack on session end, enforcing commit message standards.

MCP Protocol
Model Context Protocol — the open standard for connecting external tool servers to Claude Code.

What MCP is

MCP (Model Context Protocol) is an open standard that lets external servers expose tools to Claude Code. Claude Code acts as the MCP client; external services act as MCP servers.

Examples: a database query server, a Jira integration, a custom internal API — all appear in Claude Code as regular tools the model can call.

Transport types

stdio (subprocess)
Most common. Claude Code spawns the server as a child process and communicates over stdin/stdout.
Default
HTTP/SSE
Server runs independently. Claude Code connects over HTTP with Server-Sent Events for streaming.
Remote
In-process
SDK transport — server runs inside the same process. Used by plugins and SDK users.
SDK

Configuring MCP servers

// .claude/settings.json
{
  "mcpServers": {
    "my-db-tool": {
      "command": "node",
      "args": ["./db-mcp-server.js"],
      "env": { "DB_URL": "postgres://..." }
    }
  }
}
PM Insight — MCP as extensibility primitive
MCP is the key extensibility mechanism for enterprise. Teams can build internal tool servers (Jira, databases, proprietary APIs) and expose them to Claude Code without any changes to Claude Code itself. This is how Claude Code becomes a platform.

Agents & Tasks
Sub-agents, async task tracking, and multi-agent swarm/teams mode.

AgentTool — sub-agents

The AgentTool launches a new QueryEngine instance as a sub-agent. Sub-agents:

  • Have their own conversation context (don't see parent's messages)
  • Can be given a subset of tools (e.g., read-only tools for research)
  • Run to completion and return a single result to the parent
  • Can be run in parallel with other sub-agents
  • Optionally use isolation: "worktree" for Git worktree isolation

Task types

TypeWhat it is
local_bashA background shell command tracked as a task
local_agentA sub-agent running in the same process
remote_agentAn agent running on a remote machine/service
in_process_teammateA teammate agent in swarm/teams mode
local_workflowA multi-step workflow running locally
dreamBackground memory consolidation (autoDream)

Swarm / Teams mode

Multi-agent collaboration where a coordinator orchestrates multiple teammate agents:

  • Coordinator assigns work, collects results, synthesizes output
  • Teammates run in parallel on different subtasks
  • SendMessageTool — send messages between agents
  • RemoteTriggerTool — trigger remote agent workflows
PM Insight
AgentTool is the mechanism behind Explore, Plan, and other specialized agents. Sub-agents let the model "parallelize" work — multiple independent research tasks running simultaneously, each with a fresh context window.

PM Guide & Token Tips
How to use Claude Code effectively as a PM, and how to control API costs.

What Claude Code is good at for PMs

🔍
Codebase understanding
"How does the permissions system work?" → Claude reads the files and explains it. Far faster than asking an engineer.
📊
Feature impact analysis
"Which files would be affected by adding a new permission mode?" → Claude traces code paths and gives scope estimates.
📝
Spec validation
"Does this spec match what the code actually does?" → Claude compares your PRD against the implementation.
🗺️
Architecture tours
"Walk me through the data flow for an MCP tool call" → Claude traces through the code step by step.
🐛
Bug triage help
"What could cause a permission to be silently denied?" → Claude analyzes the system and lists plausible causes.
📋
Changelog generation
"Summarize this sprint's commits in user-facing terms" → Claude reads git log and writes the changelog.

Token cost: what drives it

Tokens = money. Context = tokens.
Every API call sends the full conversation history. The longer the session, the more tokens each message costs. Tool results (large file reads) multiply costs fast.
ActionToken costNotes
Reading a small file (50 lines)Low (~200 tok)Fine
Reading a large file (1000+ lines)High (~4000 tok)Prefer targeted reads
Bash command with short outputLowFine
Bash command with verbose logsVery highPipe through head/grep
Each web fetchHigh (~2000-8000 tok)Use sparingly
Long conversation (50+ turns)Very high baselineUse /compact or start fresh
Sub-agent launchNew context windowAgents don't inherit parent context

Token-saving strategies

1. Use /compact early and often

Type /compact to summarize old conversation into a condensed context. Cuts token count by 60-80% for long sessions. Do this after completing a major subtask.

2. Be specific in your prompts

❌ "How does authentication work?"
   → Claude reads 10+ files

✅ "How does OAuth config work in services/oauth/ only?"
   → Claude reads 2-3 targeted files

3. Use Plan mode for exploration

Start with /plan (read-only). Claude can explore without executing — faster, cheaper, safer. Switch to full mode only when ready to make changes.

4. Use sub-agents for parallel research

Sub-agents start fresh context windows. For broad research, having Claude launch parallel sub-agents is cheaper than one long session — each has a clean slate.

5. Use /clear for fresh starts

/clear resets the conversation entirely. Use between completely different tasks — don't let one session balloon across 5 different topics.

CLAUDE.md — your most powerful PM tool

# CLAUDE.md example
This is Claude Code, Anthropic's CLI.

## Rules
- Never run destructive bash commands without confirmation
- When explaining architecture, include code file references
- All new features need tests

## Project structure
- src/tools/ — all tool implementations
- Entry point is main.tsx, not index.ts

A CLAUDE.md at the project root is injected into every conversation's system prompt. Use it to set scope, enforce standards, document conventions, and restrict dangerous behaviors — permanently, without having to repeat instructions every session.

Common PM workflows

Sprint kickoff

  1. Ask Claude to read the spec/tickets
  2. "Which files will this change touch? Estimate complexity."
  3. "Are there edge cases in the current implementation that conflict with this spec?"

Bug investigation

  1. "Reproduce this bug scenario in the code — trace the path."
  2. "What's the blast radius of fixing it? What else might break?"
  3. "Write a failing test that demonstrates the bug."

Release preparation

  1. "Summarize all commits since the last tag in user-facing terms."
  2. "Are there any API-breaking changes?"
  3. "Generate a changelog entry for the release notes."
